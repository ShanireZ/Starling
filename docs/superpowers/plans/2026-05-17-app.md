Widget tasks use widget_test as the TDD analog.

**Goal:** Deliver a Flutter Android app (Android-only in v1; iOS deferred to v2 per spec §6.4) that (a) connects to the ESP32-S3 over BLE for 10 Hz live telemetry and JSON commands per IC-2, (b) downloads `.starlog.gz` log files over WiFi STA and parses them byte-for-byte identically to the firmware writer per IC-3, (c) edits and validates the **10-state** lookup table per IC-4 with a sandbox simulation that replays a historical session against the proposed table, (d) integrates with the Starling Firebase backend (Auth + Firestore + Storage + FCM + Crashlytics) per IC-5 — as a **user client** that uploads to Storage and listens to Firestore for the resulting session doc (never writes session docs directly), and (e) ships as a debug + release-signed APK installable on the user's personal Android device for self-beta.

**Architecture:** Flutter 3.x app organized into 4 layers — **models** (immutable Dart classes mirroring IC-3/IC-4/IC-5), **services** (BLE client, WiFi HTTP client, starlog parser, Firebase repos, local caches), **state** (Riverpod providers — `StateNotifier` for connection / recording state, `StreamProvider` for telemetry / Firestore listeners), and **pages** (7 routed pages, each consuming providers via `ConsumerWidget`). The BLE telemetry frame parser (50 B per IC-2) and the starlog binary parser (128 B per IC-3) are pure Dart with no Flutter dependency so they unit-test on the Dart VM. The starlog parser is golden-fixture-locked to the firmware writer and the cloud parser (same `golden_v1.starlog` fixture used in firmware plan Task 45 and cloud plan Task 10). The app **never** writes `sessions/{session_id}` Firestore docs directly — it uploads `.starlog.gz` to `gs://<bucket>/sessions/<uid>_<timestamp>.starlog(.gz)` per OQ-CLOUD-Admin (2026-05-28) and then listens to its own user's sessions collection for the doc the Cloud Function creates.

**Tech Stack:**
- Flutter 3.x + Dart 3.x (null-safe, sound mode)
- State management: **Riverpod 2.x** (`flutter_riverpod`, `riverpod_annotation`, `riverpod_generator` for `@riverpod` codegen)
- Routing: **go_router 14.x** (declarative, type-safe)
- BLE client: **flutter_blue_plus 1.32.x** (best Android BLE support; native iOS path exists for v2)
- Charts: **fl_chart 0.68.x** (real-time + offline)
- Firebase: `firebase_core`, `firebase_auth`, `cloud_firestore`, `firebase_storage`, `firebase_crashlytics`, `firebase_messaging` (all `^3.x` / aligned versions via `firebase-bom`-equivalent)
- HTTP for WiFi STA: **dio 5.x**
- Local storage: **sqflite 2.x** (relational session metadata cache) + **Hive 2.x** (key-value app preferences)
- Archive: **archive 3.x** (gzip decode for downloaded `.starlog.gz`)
- File system: **path_provider 2.x**
- Tests: **flutter_test** (built-in) + **mocktail 1.x** for mocks
- Lints: **flutter_lints 4.x** (Dart `analyzer` configured via `analysis_options.yaml`)
- CI: GitHub Actions on push (`ubuntu-latest`, `subosito/flutter-action@v2`)
- Min Android SDK 24 (Android 7.0) — chosen for `flutter_blue_plus` requirements + ESP32-S3 BLE 5.0 compatibility; target SDK 34 (Android 14).

### IC-3 — Binary log frame (this app downloads `.starlog.gz` over WiFi STA and parses it)

Per master IC-3 (OQ-3 resolved 2026-05-24): **128 bytes/frame, packed, little-endian**; CRC16 CCITT (init `0xFFFF`, poly `0x1021`) in the **last 2 bytes** (offset 126); `lookup_table_version` is **uint8 at byte 99**. The file starts with a **4 KB header** (spec §6.3 + firmware plan Task 22) carrying magic `0x53544152` ('STAR'), schema version, firmware version, bike profile, table version, start time, rider note.

```
file layout = [4096-byte header] [128-byte frame] [128-byte frame] ...

frame key offsets the app actually reads (full field list mirrors cloud Task 10):
  off 0   uint64_t  time_us
  off 8   uint32_t  gps_time_unix
  off 12  int32_t   gps_lat_e7
  off 16  int32_t   gps_lon_e7
  off 22  uint16_t  gps_speed_kmh_cx100
  off 24  uint8_t   gps_fix_type
  off 56  int16_t   roll_cdeg
  off 58  int16_t   pitch_cdeg
  off 62  int16_t   linear_accel_long_cms2
  off 64  uint16_t  wheel_speed_kmh_cx100
  off 67  uint8_t   state_current
  off 88  uint8_t   fault_flag
  off 99  uint8_t   lookup_table_version   ← parser selector
  off 126 uint16_t  crc16
```

The app's Dart parser MUST be **byte-for-byte identical** to the firmware writer (`firmware/components/log/log_frame.c`, firmware plan Task 20) and the cloud Python parser (`cloud/functions/parsers/starlog.py`, cloud plan Task 10). Strategy: share the **same** `golden_frame.bin` + `golden_v1.starlog` fixtures (produced by firmware plan Task 50) as immutable test inputs.

### IC-4 — Lookup table CSV (the table editor page reads, validates, simulates, and writes this)

> **State enum (OQ-FSM, 2026-05-28) — canonical 10 states (NOT 9):**
>
> ```
> PARKED, LOW_SPEED, CRUISE, DRAG_REDUCE, BRAKING_LIGHT,
> BRAKING_HARD, TRAIL_BRAKE, CORNERING, WHEELIE_GUARD, FAULT
> ```
>
> Numeric values strictly in declaration order (PARKED=0, LOW_SPEED=1, ..., FAULT=9). The legacy "9-state" wording in spec §5.2 title and older READMEs is non-authoritative documentation drift — any new code, model, test fixture, or UI element in this plan MUST handle **10** values.

```csv
# Starling lookup table v1
# state, speed_kmh, angle_deg
PARKED,0,0
LOW_SPEED,*,0           # * = wildcard, all speeds
CRUISE,*,0
DRAG_REDUCE,150,-3
DRAG_REDUCE,170,-4
DRAG_REDUCE,195,-5
BRAKING_LIGHT,30,25
BRAKING_LIGHT,50,40
BRAKING_LIGHT,80,55
BRAKING_LIGHT,120,60
BRAKING_LIGHT,160,62
BRAKING_HARD,30,30
BRAKING_HARD,50,50
BRAKING_HARD,80,65
BRAKING_HARD,120,70
BRAKING_HARD,160,70
TRAIL_BRAKE,*,(0.5 * BRAKING_LIGHT(speed))
CORNERING,*,-3
WHEELIE_GUARD,*,30
FAULT,*,0

# Metadata
# bike_profile: gsx250r | rc450 | universal
# table_version: 1
# created: 2026-05-17T00:00:00Z
# author: shanire
# crc16: 0xABCD  (over all rows excluding this line)
```

Interpolation rules (IC-4):
- Per state, sort rows by speed; piecewise-linear between rows.
- Below the lowest speed: clamp to lowest row's angle.
- Above the highest speed: clamp to highest row's angle.
- `*` wildcard: state has one constant angle for all speeds (and may not be mixed with non-wildcard rows in the same state).
- `TRAIL_BRAKE` is an expression that delegates to `BRAKING_LIGHT(speed) * 0.5`.

Angle physical range (OQ-7, 2026-05-24): **−5° … +70°** total travel. Spring-neutral / fail-safe / FAULT target = 0° (flush). −5° is powered-only (servo actively drives below neutral).

### IC-5 — Firebase data model (this app is a **user client**, not admin)

```
Firestore collections:
  users/{user_id}              email, created_at, bikes: [bike_id]
  bikes/{bike_id}              user_id, profile: "gsx250r" | "rc450", vin?, created_at, last_session_id
  sessions/{session_id}        user_id, bike_id, start_time, duration_sec, distance_km,
                               max_speed_kmh, fault_count,
                               state_dwell_percent: {<10-state enum keys>: <int>},
                               storage_uri: "gs://bucket/sessions/<session_id>.starlog",
                               firmware_version, table_version,
                               analyzed_at?: timestamp,
                               summary?: {braking_events: [...], peak_decel: ...}
  firmware_releases/{version}  version, bike_profile: "gsx250r"|"rc450"|"universal" (OQ-CLOUD-B),
                               storage_uri, rollout_percentage, released_at, changelog

Cloud Storage paths:
  /sessions/<session_id>.starlog                       (canonical, written by Cloud Function)
  /sessions/<uid>_<timestamp>.starlog(.gz)             (client upload path — what THIS APP writes)
  /firmware/<version>.bin
  /lookup_tables/<bike_profile>/<version>.csv

Cloud Functions used by this app:
  - on_session_upload()  : Storage trigger — app uploads, function writes Firestore doc
  - get_signed_url()     : HTTPS callable — owner-only signed URL for session download
```

**App's allowed operations** (Auth role: signed-in user, NO admin claim):
- `users/{auth.uid}` — read + write own doc
- `bikes/{bikeId}` where `user_id == auth.uid` — full CRUD
- `sessions/{sessionId}` — **read only** (writes via Cloud Function — OQ-CLOUD-Admin 2026-05-28); app **MUST NOT** call `.set()` / `.update()` on session docs
- `firmware_releases/{version}` — read only (anonymous-readable per IC-5)
- Storage `sessions/<auth.uid>_*.starlog(.gz)` — write + read (per cloud plan Task 6 rule)
- Storage `firmware/<version>.bin` — read only
- Storage `lookup_tables/<profile>/<version>.csv` — read only

**Session upload pattern (OQ-CLOUD-Admin, 2026-05-28):**
1. App downloads `.starlog` from ESP32 via WiFi STA.
2. App gzips locally → `.starlog.gz`.
3. App uploads to `gs://<bucket>/sessions/<auth.uid>_<YYYYMMDD_HHMMSS>.starlog.gz` (the `<uid>_` prefix is enforced by storage rules — see cloud plan Task 6).
4. Cloud Function `on_session_upload` fires on Storage finalize, parses the file, renames to canonical `sessions/<session_id>.starlog.gz`, writes Firestore `sessions/{session_id}` doc.
5. App's Firestore listener on `sessions where user_id == auth.uid orderBy start_time desc` sees the new doc and refreshes the UI.

**FCM topic subscription (OQ-CLOUD-B, 2026-05-28):**
- When user signs in or changes their current bike's `profile` in Settings, the app subscribes to FCM topic **`firmware-<bike_profile>`** AND **`firmware-universal`**, unsubscribing from the previous `firmware-<old_profile>` if the profile changed.
- Topic names use exactly the IC-5 bike_profile values: `firmware-gsx250r`, `firmware-rc450`, `firmware-universal`.
- The data-only FCM payload (per cloud plan Task 14 + IC-6) carries `{op: "firmware_available", version, bike_profile, changelog}`. The app shows an in-app banner; the firmware decides when to download via WiFi STA (the firmware, not the app, is the OTA downloader).
>
> **NOTE: app↔cloud FCM topic format inconsistency.** Cloud plan Task 14 currently uses **per-user** topics (`bike-<profile>-<user_id>`) computed server-side from the bikes collection. This master-plan amendment (OQ-CLOUD-B 2026-05-28) and this app plan use **broadcast** topics `firmware-<bike_profile>` + `firmware-universal` that the client subscribes to itself (no server-side bike lookup needed). This is **simpler**, **scales better**, and is what this app plan is built against. **The cloud plan Task 14 will need revision to match** (replace `eligible_user_topics` with `messaging.send(Message(topic=f"firmware-{bike_profile}"))` and a second send for `firmware-universal`). This app plan flags the divergence rather than re-deriving from the older cloud handler. See "Open Items" at the end of this plan.

### IC-6 — Power budget (not directly applicable to phone app)

The app does not power-constrain the firmware. The only downstream effect: when the app issues `start_recording` it should not flood BLE writes; the firmware staggers its own servo activity. The app rate-limits its outbound JSON commands to ≤ 2 Hz (Task 13).

```dart
import 'package:flutter/material.dart';

/// Single source of truth for app colors. Numeric values are hand-picked
/// for sufficient contrast on AMOLED Android screens in bright daylight
/// (this is a motorcycle app — sunlight readability matters).
class SemanticColors {
  const SemanticColors();
  Color get parked => const Color(0xFF607D8B);          // blue-grey
  Color get lowSpeed => const Color(0xFF8D6E63);        // brown
  Color get cruise => const Color(0xFF43A047);          // green
  Color get dragReduce => const Color(0xFF00897B);      // teal
  Color get brakingLight => const Color(0xFFFB8C00);    // orange
  Color get brakingHard => const Color(0xFFE53935);     // red
  Color get trailBrake => const Color(0xFFAB47BC);      // purple
  Color get cornering => const Color(0xFF1E88E5);       // blue
  Color get wheelieGuard => const Color(0xFFFFA000);    // amber
  Color get fault => const Color(0xFFD32F2F);           // dark red
}

const semanticColors = SemanticColors();

ThemeData starlingTheme() {
  final scheme = ColorScheme.fromSeed(
    seedColor: const Color(0xFF1E88E5),
    primary: const Color(0xFF1E88E5),
    brightness: Brightness.light,
  );
  return ThemeData(
    useMaterial3: true,
    colorScheme: scheme,
    textTheme: const TextTheme().apply(fontFamily: 'Inter'),
    appBarTheme: AppBarTheme(
      backgroundColor: scheme.primary,
      foregroundColor: scheme.onPrimary,
      centerTitle: false,
    ),
  );
}
```
