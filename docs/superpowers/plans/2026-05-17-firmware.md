**Goal:** Deliver ESP-IDF C/C++17 firmware for the ESP32-S3 controller implementing a 9-state priority-arbitrated control loop at 100 Hz with closed-loop position PID and lookup-table fallback ("Èó≠ÁéØ PIDÔºå‰ΩÜÊúâÊü•Ë°®ÊâòÂ∫ï"), plus a separate ~50-line ATtiny85 watchdog firmware, plus the `starlog_analyzer` Python package for offline log analysis. The entire control loop is bench-testable with **no physical wing/servo/bike** via per-driver STUB modes.

**Architecture:** A layered ESP-IDF component build (Layer 0 HAL ‚Üí Layer 1 fusion ‚Üí Layer 2 control ‚Üí Layer 3 logging ‚Üí Layer 4 comm/OTA), driven by FreeRTOS tasks. A 100 Hz timer-driven control task is the hot path: it must contain **no exceptions and no heap-allocating STL containers**. A separate ATtiny85 firmware (AVR-GCC, not ESP-IDF) provides the independent hardware watchdog (safety Layer 3 of 5). A Python package decodes the binary `.starlog` files byte-for-byte identically to the firmware writer.

**Tech Stack:**
- ESP-IDF v5.x (Espressif official), `idf.py` build system, target `esp32s3`
- C / C++17 ‚Äî **no exceptions, no STL containers in the 100 Hz hot path** (static buffers, ring buffers, plain structs)
- FreeRTOS (bundled with ESP-IDF) ‚Äî pinned tasks, `esp_timer` high-resolution periodic callback for the control loop
- BLE: **NimBLE host** (NOT Bluedroid) ‚Äî lower RAM, the IDF-recommended host
- Wi-Fi STA: ESP-IDF `esp_wifi` + `esp_http_server`
- OTA: `esp_https_ota` + dual-bank (`factory` + `ota_0` + `ota_1`) partition table, app rollback enabled
- Unit tests: **Unity** (ESP-IDF built-in). Logic-only modules (FSM, table parser, CRC, frame layout, fusion math) are compiled and run on the **Linux host target** (`idf.py --preview set-target linux`) so tests run in CI with no hardware. Hardware-touching tests run on-device via `pytest-embedded`.
- ATtiny85: AVR-GCC + `avrdude` (separate sub-project, NOT ESP-IDF)
- Python analyzer: numpy + pandas + matplotlib + pytest, packaged with `setup.py` / `pyproject.toml`

```c
#define GPIO_USER_BUTTON        0    // boot/calibration trigger button
// GPIO 22-48: reserved/unused. Former HX711 pins (20/21) are NOT used for strain gauge ‚Äî
//             strain gauge is a STANDALONE BLE module, OUT OF SCOPE here.

// I¬≤C addresses (one shared bus on SDA=8 / SCL=9)
#define I2C_ADDR_IMU_BMI270     0x68
#define I2C_ADDR_AS5600L_LEFT   0x40   // AS5600L (WITH the L), ADDR pin = GND
#define I2C_ADDR_AS5600L_RIGHT  0x41   // AS5600L (WITH the L), ADDR pin = VCC
```

> **CRITICAL:** GPIO 19/20 are native USB only ‚Äî no driver may claim them. The FAULT status LED is GPIO 21 (`FAULT_STATUS_OUT`), never 19. The encoders are **AS5600L** (with the L) ‚Äî the plain AS5600 has a fixed 0x36 address and cannot share a bus; do not use AS5600 register quirks blindly, but the L variant is register-compatible for the angle registers used here.

### IC-3 ‚Äî Binary SD log frame (verbatim; full struct in spec ¬ß6.2)

```
- 128 bytes/frame, packed, little-endian
- CRC16 CCITT (init 0xFFFF, poly 0x1021) in the LAST 2 bytes
- byte offset 99 = lookup_table_version (Python analyzer selects parser by this)
- written frame-by-frame sequentially to SD
- 4 KB file header with session metadata precedes the frames
```

Full struct (spec ¬ß6.2), reproduced for the writer:

```c
typedef struct __attribute__((packed)) {
    // time (12B)
    uint64_t time_us;                 // off 0
    uint32_t gps_time_unix;           // off 8
    // GPS (24B)
    int32_t  gps_lat_e7;              // off 12
    int32_t  gps_lon_e7;              // off 16
    int16_t  gps_alt_dm;              // off 20
    uint16_t gps_speed_kmh_cx100;     // off 22
    uint8_t  gps_fix_type;            // off 24
    uint8_t  gps_hdop;                // off 25
    uint8_t  gps_sats;                // off 26
    uint8_t  pad1[5];                 // off 27..31
    // IMU raw (24B)
    int16_t  ax, ay, az, gx, gy, gz;  // off 32..43
    uint8_t  pad2[12];                // off 44..55
    // fused attitude (8B)
    int16_t  roll_cdeg;               // off 56
    int16_t  pitch_cdeg;              // off 58
    int16_t  yaw_cdeg;                // off 60
    int16_t  linear_accel_long_cms2;  // off 62
    // control (24B)
    uint16_t wheel_speed_kmh_cx100;   // off 64
    uint8_t  brake_lever_active;      // off 66
    uint8_t  state_current;           // off 67
    int16_t  target_angle_l_cdeg;     // off 68
    int16_t  target_angle_r_cdeg;     // off 70
    int16_t  measured_angle_l_cdeg;   // off 72
    int16_t  measured_angle_r_cdeg;   // off 74
    uint16_t servo_current_l_ma;      // off 76
    uint16_t servo_current_r_ma;      // off 78
    uint8_t  pad3[8];                 // off 80..87
    // health (8B)
    uint8_t  fault_flag;              // off 88
    uint8_t  health_bits;             // off 89
    uint16_t supply_voltage_mv;       // off 90
    uint16_t supercap_voltage_mv;     // off 92
    uint16_t cpu_loop_us;             // off 94
    // misc (30B) ‚Äî CANONICAL layout per IC-3 (OQ-3 RESOLVED): version is uint8 @ byte 99.
    uint8_t  pad_misc[3];             // off 96..98  (aligns version to byte 99)
    uint8_t  lookup_table_version;    // off 99  (IC-3 invariant: byte 99, uint8)
    uint8_t  pad4[26];                // off 100..125
    // CRC (2B)
    uint16_t crc16;                   // off 126
} starlog_frame_t;                    // total 128 bytes
```

> **OQ-3 RESOLVED (2026-05-24):** IC-3 is authoritative ‚Äî `lookup_table_version` is a **uint8 at byte 99**, frame is **128 B**, CRC16 CCITT @ **offset 126**. The earlier spec ¬ß6.2 struct was malformed (its field ordering summed to 124 B with the version landing at offset 96); **spec ¬ß6.2 has been patched to match IC-3** (see spec ¬ß6.2 AMENDMENT note). This firmware layout already implements IC-3 correctly ‚Äî no change needed beyond closing the open question. Task 20 pins every offset via `offsetof`/`_Static_assert`, and the Python analyzer (Task 44) decodes the identical layout (`TABLE_VERSION_OFFSET=99`).

### IC-4 ‚Äî Lookup table CSV (verbatim)

Piecewise-linear interpolation over `speed_kmh`; `*` wildcard ignores speed; expression rows reference other states, e.g. `TRAIL_BRAKE,*,(0.5 * BRAKING_LIGHT(speed))`. Interpolation rules: clamp below min row, clamp above max row, linear between. This parser is the lookup-table fallback ("Êü•Ë°®ÊâòÂ∫ï") that sits beneath the closed-loop PID.

### IC-5 ‚Äî Firebase (firmware cares only about OTA)

OTA client polls Cloud Storage `firmware/<version>.bin`; release metadata at Firestore `firmware_releases/{version}` with `version`, `storage_uri`, `rollout_percentage`, `released_at`, `changelog`. The firmware fetches a signed URL (Cloud Function `get_signed_url`) and streams the `.bin` to the inactive OTA partition.

### IC-6 ‚Äî Power budget (hard firmware correctness requirement)

USB-PD ceiling 9 V @ 5 A = 45 W. Both servos at full draw = 50 W > 45 W. **Therefore the servo driver MUST stagger left/right servo motion by 50 ms** so the two never peak simultaneously. Each servo ‚â§ 30 W continuous. This is asserted in a unit test (Task 7).

`crc16.c`:

```c
#include "crc16.h"
uint16_t crc16_ccitt(const uint8_t *data, size_t len) {
    uint16_t crc = 0xFFFF;
    for (size_t i = 0; i < len; i++) {
        crc ^= (uint16_t)data[i] << 8;
        for (int b = 0; b < 8; b++)
            crc = (crc & 0x8000) ? (uint16_t)((crc << 1) ^ 0x1021) : (uint16_t)(crc << 1);
    }
    return crc;
}
```

```python
def crc16_ccitt(data: bytes) -> int:
    crc = 0xFFFF
    for b in data:
        crc ^= b << 8
        for _ in range(8):
            crc = ((crc << 1) ^ 0x1021) & 0xFFFF if (crc & 0x8000) else (crc << 1) & 0xFFFF
    return crc
```
